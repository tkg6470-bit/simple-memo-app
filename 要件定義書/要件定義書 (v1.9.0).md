\# **\*\*📋 要件定義書 (System Requirements Specification)\*\***

Project Name: Simple Memo App (Enhanced Edition)

Version: 1.9.0 (CI/CD Enabled Edition)

本ドキュメントは、モダンな自社開発企業が採用する技術スタックと開発フローを再現した、フルスタックメモアプリケーションの要件を定義するものである。  
v1.9.0では、コード品質の担保とデプロイの安全性を高めるため、**\*\*GitHub Actions による CI/CD パイプライン\*\*** を正式な開発基盤として組み込んだ。

\#\# **\*\*3\\. 非機能要件 (Non-Functional Requirements)\*\***

\#\#\# **\*\*3.1 セキュリティ & インフラ\*\***

\* **\*\*機密情報\*\***: .env 管理を徹底。Render Dashboard 上で環境変数を注入する。  
\* **\*\*入力値検証\*\***: Zod による厳格なバリデーションを実施。  
\* **\*\*可用性\*\***:  
  \* **\*\*Render\*\***: Docker ベースの Web Service。  
  \* **\*\*S3互換性\*\***: Supabase Storage 接続時は \`forcePathStyle: true\` を必須とする。

\#\#\# **\*\*3.2 品質・保守性\*\***

\* **\*\*型安全性\*\***: TypeScript を全面採用し、\`any\` 型の使用を原則禁止する（Hono RPC導入までは一部許容）。  
\* **\*\*自動化 (CI)\*\***: GitHub Actions により、プッシュのたびに Lint と Type Check を自動実行する。  
\* **\*\*依存関係\*\***: 本番環境では \`npm ci \--omit=dev\` を使用し、不要なパッケージを含めない。  
\* **\*\*アーキテクチャ\*\***: 「関心の分離」に基づき、ルーティング、コントローラー、サービス層を分離する。

\#\#\# **\*\*3.3 パフォーマンス目標\*\***

\* **\*\*API レスポンス\*\***: 通常の CRUD 操作は **\*\*500ms 以内\*\*** にレスポンスを返すこと。  
\* **\*\*AI 処理\*\***: 時間のかかる AI 処理（数秒〜）は非同期で行い、UI をブロックしないこと。

\#\# **\*\*6\\. 開発運用ルール (Development & Operations Rules)\*\***

\#\#\# **\*\*6.1 環境分離 (Environment Isolation)\*\***

\* **\*\*Dev Containers\*\***: すべての開発は Docker コンテナ内で行う。  
\* **\*\*Frontend 接続\*\***: 環境変数 \`VITE\_API\_BASE\_URL\` を使用し、ローカル開発時と本番（Render）で接続先を切り替える。

\#\#\# **\*\*6.2 ビルド戦略 (Build Strategy) \- Render最適化\*\***

Render Free Tier のメモリ制限 (512MB) 回避のため、以下の戦略をとる。

1\.  **\*\*ローカルビルド\*\***: 開発機で \`npm run build\` を実行し、\`dist\` を生成。  
2\.  **\*\*アーティファクト管理\*\***: \`dist\` を Git リポジトリに含めてプッシュする。  
3\.  **\*\*Render 実行\*\***: 起動コマンド \`node dist/index.js\` のみを実行する。

\#\#\# **\*\*6.3 バージョン管理・ブランチ戦略\*\***

\* **\*\*フロー\*\***: GitHub Flow (main \\+ feature ブランチ)。  
\* **\*\*コミットメッセージ\*\***: Feat:, Fix:, Refactor:, Docs: 等のプレフィックスを付与する。

\#\#\# **\*\*6.4 エラーハンドリング設計\*\***

| コード | 意味 | フロントエンドの挙動 |  
| :---- | :---- | :---- |  
| **\*\*200/201\*\*** | OK/Created | 成功メッセージを表示または画面遷移。 |  
| **\*\*400\*\*** | Bad Request | 入力エラー箇所をハイライト表示。 |  
| **\*\*401\*\*** | Unauthorized | ログイン画面へリダイレクト。 |  
| **\*\*500\*\*** | Internal Error | 「システムエラーが発生しました」とトースト表示。 |

\#\#\# **\*\*6.5 CI/CD パイプライン (New)\*\***

品質を担保するため、GitHub Actions による以下の自動チェックを導入する。

\* **\*\*トリガー\*\***: \`main\` ブランチへの Push および Pull Request。  
\* **\*\*チェック項目\*\***:  
    1\.  **\*\*Backend\*\***: \`npm run lint\` (ESLint), \`npm run typecheck\` (tsc)  
    2\.  **\*\*Frontend\*\***: \`npm run lint\` (ESLint), \`npm run typecheck\` (tsc)  
\* **\*\*運用ルール\*\***:  
    \* 開発者はプッシュ前に必ずローカルで \`npm run lint\` 等を実行し、エラーがないことを確認する。  
    \* **\*\*CI が「Green (Success)」の状態でのみ、Render へのデプロイが正当なものとみなされる。\*\***

\#\# **\*\*7\\. 制約事項・前提条件 (Constraints)\*\***

\* **\*\*ブラウザサポート\*\***: Google Chrome, Safari, Edge, Firefox の最新版。  
\* **\*\*言語対応\*\***: 日本語のみ。  
\* **\*\*依存サービス\*\***: OpenAI API 等の外部サービス停止時は Graceful Degradation させる。
