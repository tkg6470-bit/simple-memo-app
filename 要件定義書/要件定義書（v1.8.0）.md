**📋 要件定義書 (System Requirements Specification)**

Project Name: Simple Memo App (Enhanced Edition)

Version: 1.8.0 (Render & Supabase Edition)

本ドキュメントは、モダンな自社開発企業が採用する技術スタックと開発フローを再現した、フルスタックメモアプリケーションの要件を定義するものである。  
v1.8.0では、運用コストと保守性を最適化するため、インフラ基盤を AWS から Render & Supabase へ移行した構成とする。

## **1\. プロジェクト概要**

### **1.1 目的**

* **技術力の証明**: 未経験から即戦力レベルのモダン技術（TypeScript, Docker, Cloud-Native, AI 連携）に加え、**PaaS/BaaS の特性を理解した環境構築能力**を証明する。

### **1.2 予算戦略**

* **開発・運用フェーズ**: 完全無料  
  * **Compute**: Render (Free Tier) \- Docker コンテナ実行  
  * **Database**: Supabase (Free Tier) \- PostgreSQL  
  * **Storage**: Supabase Storage (Free Tier) \- 画像保存  
  * **Auth**: Clerk (Free Tier)  
* **備考**: AWS (App Runner/RDS) 環境は設計資産として保持するが、当面はコスト削減のため凍結する。

## **3\. 非機能要件 (Non-Functional Requirements)**

### **3.1 セキュリティ & インフラ**

* **機密情報**: .env 管理を徹底。Render Dashboard 上で環境変数を注入する。  
* **入力値検証**: Zod による厳格なバリデーションを実施。  
* **可用性**:  
  * **Render**: Docker ベースの Web Service。  
  * **Supabase**: フルマネージド PostgreSQL \+ S3 互換ストレージ。  
  * **S3互換性**: Supabase Storage 接続時は forcePathStyle: true を必須とする。

### **3.2 品質・保守性**

* **型安全性**: TypeScript を全面採用し、any 型の使用を原則禁止する。  
* **依存関係**: 本番環境では npm ci \--omit=dev を使用し、不要なパッケージを含めない。  
* **ライブラリ**: Prisma 実行に必要な openssl をコンテナ内に明示的にインストールする。  
* **アーキテクチャ**: 「関心の分離」に基づき、ルーティング、コントローラー、サービス層を分離する。  
* **テスト**: Vitest (単体) / Playwright (E2E) の実装。

### **3.3 パフォーマンス目標**

* **API レスポンス**: 通常の CRUD 操作は **500ms 以内** にレスポンスを返すこと。  
* **AI 処理**: 時間のかかる AI 処理（数秒〜）は非同期で行い、UI をブロックしないこと。

## **6\. 開発運用ルール (Development & Operations Rules)**

### **6.1 環境分離 (Environment Isolation)**

* **Dev Containers**: すべての開発は Docker コンテナ内で行う。  
* **Frontend 接続**: 環境変数 VITE\_API\_BASE\_URL を使用し、ローカル開発時と本番（Render）で接続先を切り替える。  
* **VS Code**: 拡張機能（ESLint, Prettier 等）もコンテナ内に定義し、開発者間で環境差異が生まれないようにする。

### **6.2 ビルド戦略 (Build Strategy) \- Render最適化**

Render Free Tier のメモリ制限 (512MB) によるビルド落ち (OOM) を回避するため、以下の戦略をとる。

1. **ローカルビルド**: 開発機（Dev Container）で npm run build を実行し、dist ディレクトリを生成する。  
2. **アーティファクト管理**: dist ディレクトリを .gitignore から外し、Git リポジトリに含めてプッシュする。  
3. **Render 実行**: Render 上ではビルドコマンドを空（または最小限）にし、起動コマンド node dist/index.js のみを実行する。  
4. **軽量コンテナ**: node:20-slim をベースにし、必要なライブラリ (openssl) のみを追加インストールする。

### **6.3 バージョン管理・ブランチ戦略**

* **フロー**: GitHub Flow (main \+ feature ブランチ)。  
* **コミットメッセージ**: Feat:, Fix:, Refactor:, Docs: 等のプレフィックスを付与する。

### **6.4 エラーハンドリング設計**

API は以下の HTTP ステータスコードを適切に返却し、フロントエンドはこれに基づいて UI を制御する。

| コード | 意味 | フロントエンドの挙動 |
| :---- | :---- | :---- |
| **200/201** | OK/Created | 成功メッセージを表示または画面遷移。 |
| **400** | Bad Request | 入力エラー箇所をハイライト表示（Zod 連携）。 |
| **401** | Unauthorized | ログイン画面へリダイレクト。 |
| **404** | Not Found | 404 ページまたは一覧へリダイレクト。 |
| **500** | Internal Error | 「システムエラーが発生しました」とトースト表示。 |

## **7\. 制約事項・前提条件 (Constraints)**

* **ブラウザサポート**: Google Chrome, Safari, Edge, Firefox の最新版のみをサポート対象とする（IE 等は非対応）。  
* **言語対応**: 日本語のみとする。  
* **依存サービス**: OpenAI API 等の外部サービスが停止した場合、AI 機能は利用不可となるが、基本的なメモ機能は継続して利用できる設計とする（Graceful Degradation）。
